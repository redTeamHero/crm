Here’s the rewrite I’d use for Evolv so the app behaves like a strategy engine, not just a letter sequencer.

Important note: this is product and workflow guidance, not legal advice. Also, consumers can dispute credit-report errors themselves for free, and both the bureau and the furnisher can be disputed directly.  ￼

What to change first

Your current setup is not badly organized, but it has four weak spots:

1. You rely too much on generic defaults.
611-general-dispute, debt-validation, and second-round-dispute are fine as fallback templates, but they should not be the main engine for most accounts. CFPB and FTC guidance emphasize that disputes should identify each item, explain exactly what is wrong, and include supporting facts or documents. Generic loops are easier for a bureau or furnisher to process without changing anything.  ￼

2. “Method of verification” is placed too late.
Under FCRA 15 U.S.C. § 1681i(a)(7), after a reinvestigation, the consumer can request a description of the procedure used to determine the accuracy and completeness of the information. That means this should be a core round-2 tool, not mostly a post-verified afterthought.  ￼

3. Escalation is too time-based instead of evidence-based.
A CFPB complaint is best used when the consumer disputes, gets an unsatisfactory result, or the company’s handling appears deficient. CFPB explicitly routes the complaint to the company for review and response. That makes escalation more powerful when tied to a documented failure, not just “round 3 happened.”  ￼

4. Pay-for-delete appears too early.
That is a settlement tactic, not an investigation tactic. It should come only after the app concludes there is no strong inaccuracy, no meaningful reporting contradiction, and no better pressure path left. That part is a strategic recommendation from me, but it matches the broader principle that disputes should first focus on specific inaccuracies and supporting proof.  ￼

Recommended sequence model

Instead of organizing mostly by “first round / follow-up / no response / verified,” organize by:

A. What the account is
collection, charge-off, late payment, bankruptcy, identity/personal info, repo, judgment/public record, etc.

B. What the evidence says
factual error, Metro 2 inconsistency, obsolete reporting, duplicate reporting, reinserted item, missing investigation clarity, valid debt with weak deletion odds, and so on.

C. What happened last
no response, verified, updated, partial correction, deleted, reinserted.

That gives you a matrix like this:

Phase 1: Precision dispute

Use the strongest account-specific dispute first.

Phase 2: Investigation pressure

Use procedure/method-of-verification requests and furnisher-side disputes.

Phase 3: Compliance pressure

Use reinsertion, repeated inaccuracy, inconsistent reporting, documentation failure, or escalation logic.

Phase 4: Resolution path

Settlement, goodwill, or final complaint path.

Better sequencing by scenario

1) Collections

First touch

Use:
	•	debt validation only if the collection context supports it
	•	factual inaccuracy
	•	Metro 2 inconsistency if present
	•	obsolete debt if age supports it

Do not default to a vague general dispute if you already know the problem.

Round 2

If it comes back verified or unchanged:
	•	method of verification / procedural request
	•	furnisher/direct dispute
	•	demand for clarification of dates, balance, ownership, status, and account history

FTC and CFPB both say consumers can dispute with both the bureau and the furnisher, and furnishers have duties regarding accurate reporting.  ￼

Round 3

If still inaccurate:
	•	Metro 2 deletion demand
	•	CFPB complaint package
	•	AG or attorney referral workflow only if your product supports a documented compliance path

Last resort
	•	pay-for-delete
Only after the app concludes the debt is likely valid and the evidence path is weak.

⸻

2) Charge-offs / revolving accounts / installment accounts

First touch

Best defaults:
	•	factual errors
	•	balance/payment-status inconsistency
	•	date inconsistency
	•	Metro 2 inconsistency

Round 2
	•	method of verification
	•	furnisher dispute
	•	procedure request tied to exact tradeline fields

Round 3
	•	escalation if inaccurate after prior detailed disputes
	•	CFPB complaint if investigation result appears unsupported

Again, the reason to shift to detailed, itemized disputes is that regulators’ consumer guidance focuses on identifying each disputed item, explaining why it is wrong, and supporting the claim with facts/documents.  ￼

⸻

3) Late payments

Late payments are different.

First touch

Default should usually be:
	•	factual error, if provably wrong
	•	goodwill, if accurate but the consumer wants discretionary relief

Don’t do this

Do not overuse hard legal-sounding disputes on clearly accurate isolated lates with no evidence problem. That just burns credibility.

Better rule

If late payment is:
	•	clearly inaccurate → dispute
	•	accurate but consumer has strong history/hardship story → goodwill
	•	mixed or insufficient proof → gather docs first

⸻

4) Bankruptcy

First touch
	•	bankruptcy misreporting
	•	court/status mismatch
	•	reporting date/status inconsistency

Round 2
	•	method of verification / procedure request
	•	direct furnisher dispute where applicable

Escalate faster

Bankruptcy reporting errors can create major damage. If it is clearly wrong and remains wrong after a detailed dispute, move to complaint workflow sooner rather than later.

⸻

5) Reinserted item

This should be a special fast-lane, not a normal round bucket.

FCRA 15 U.S.C. § 1681i(a)(5) governs deletion and reinsertion procedures. If an item was deleted and later reappears, your app should not just send another generic dispute. It should trigger a reinsertion-specific path immediately.  ￼

Recommended flow:
	•	detect prior delete
	•	detect reappearance
	•	send reinsertion-specific dispute
	•	if unresolved, complaint workflow

So your current “reinserted / reappearing item” is right to exist, but it should be a high-priority exception route.

⸻

6) Identity theft / mixed file / wrong personal info

These should not live in the same generic queue as standard disputes.

FCRA 15 U.S.C. § 1681c-2 requires blocking information resulting from alleged identity theft if the consumer provides the required documentation.  ￼

So for identity theft:
	•	route to identity-theft block workflow
	•	require identity theft report and proof package
	•	do not send routine general disputes first if an identity-theft block is the proper path

- Avoid repetitive generic disputes.
- Escalate only when the file supports escalation.
